## 3. ORM Conventions

- **Synchronous** — all routes use `db: Session = Depends(get_db)`, not `AsyncSession`.
- Integer primary keys (`GENERATED BY DEFAULT AS IDENTITY` in PostgreSQL).
- All timestamps are `TIMESTAMPTZ`, defaulting to `now()`.
- Relationships use `Mapped[...]` typed annotations (SQLAlchemy 2.0 style).
- Cascade deletes are defined both in ORM (`cascade="all, delete-orphan"`) and in the SQL schema (`ON DELETE CASCADE`).
- Current ORM models are kept in one file because the domain is still small. Split into `src/models/` only after crawler/chat/vector metadata models are added.

## 4. Auth Implementation

### Student auth
- On signup/login, `create_access_token(user)` returns a JWT signed with a secret (from env or hardcoded — **must move to env before deploy**).
- The token encodes `student_id` as the subject.
- `get_current_user` dependency decodes the token, looks up the user, and injects the `User` ORM object.

### Admin auth
- Static token `pnu-pathfinder-admin-local-token`, hardcoded in `admin/router.py`.
- Admin credentials: `root` / `3011`, also hardcoded.
- **Must be moved to environment variables before any deployment.**

## 5. Database Migrations

No Alembic. Apply changes by running SQL files manually:

```bash
# Create the database
createdb pnu_pathfinder

# Apply initial schema
psql -d pnu_pathfinder -f backend/migrations/001_initial_schema.sql

# Apply later schema changes
psql -d pnu_pathfinder -f backend/migrations/002_course_records.sql
psql -d pnu_pathfinder -f backend/migrations/003_ingestion_documents.sql

# Optionally seed data
psql -d pnu_pathfinder -f backend/seeds/001_academic_programs.sql
```

Naming convention for new migrations: `NNN_description.sql` (zero-padded number).

## 7. Running Locally

```bash
cd backend

# Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Set up the database
createdb pnu_pathfinder
psql -d pnu_pathfinder -f migrations/001_initial_schema.sql

# Configure environment
cp .env.example .env

# Start the development server
uvicorn src.main:app --reload --port 8000
```

Interactive API docs: `http://localhost:8000/docs`

## 9. Data Ingestion Plan

Crawler output should be split by data type.

- Structured academic data goes to PostgreSQL:
  - academic programs
  - curriculum courses
  - graduation requirements
  - year/major-specific requirement rules
- Unstructured documents go to VectorDB:
  - notices
  - PDF/HTML guides
  - scholarship, internship, and recruitment documents
  - academic regulation text

Graduation calculation must use PostgreSQL, not VectorDB. Vector search is for AI context retrieval, while graduation progress is deterministic rule calculation.

Current prototype ingestion endpoint: `POST /api/admin/crawl/run`. It accepts already-collected structured program data and document text, then writes programs to PostgreSQL and document chunks to `document_chunks`.
